## User Preferences

<!-- How the user likes things done. Code style, tools, patterns, communication. -->
- **Salary model (2026-06-03):** Salary tab uses simple monthly net-entry model. No hourly/monthly type distinction. Each month user enters the net payslip amount directly (stored in `salaries.base`). External bonuses/deductions stored in `additions`/`deductions` JSONB. `total = base + additions - deductions`. Employee profile has no salary reference field.
- **Manual over automatic for money-matching (2026-07-05):** User explicitly rejected automatic FIFO reconciliation for supplier checks vs. debts — he sometimes deliberately pays a newer month's invoices while intentionally leaving an older month open (a dispute or business reason), and does NOT want the system to silently close the older debt first. Any debt/payment matching logic must let the user pick which specific debt(s)/month(s) a payment settles, with the system only suggesting a default (editable), never deciding unilaterally.
- **Debt is "settled" at check hand-over, not at due date:** From this user's POV, once he signs and hands over a check to a supplier, that debt is closed immediately — regardless of the check's future due date. The due date only matters for cash-flow forecasting (when money actually leaves the bank), never for whether a debt is considered paid.
- **Wants bulk-entry tools for repetitive data:** e.g. generating a whole series of checks (sequential check numbers + dates) in one action instead of typing each one — look for opportunities to batch near-identical manual entry across the app.
- **Prefers Hebrew replies mid-session when asked** — user asked explicitly ("בעברית בבקשה") partway through a conversation that started in Hebrew anyway; treat this as a durable preference for this project.

## Key Learnings

- **Project:** autoline-app
- **Description:** This is a [Next.js](https://nextjs.org) project bootstrapped with [`create-next-app`](https://nextjs.org/docs/app/api-reference/cli/create-next-app).
- **Role check pattern:** `isAdmin = profile.role === 'admin' || profile.role === 'super_admin'`. Non-admin employees should NOT see: AlertsPanel, RemindersPanel (dashboard), movements tab in Tires. Employee view in /employees shows only a phone-book (no salary data).
- **Tires permissions:** `isAdmin` controls access to the movements tab. `viewOnly` = has `tires_view` module but not `tires` and not admin (read-only inventory). Employees with neither module see nothing.
- **Shared profile/tenant context (2026-06-16):** `lib/contexts/ProfileContext.tsx` (`ProfileProvider`/`useProfile()`) fetches session+profile+full tenant row ONCE per page load. It's wired into `AppShell` (wraps Header, Sidebar, and the page's children). Any component rendered as a descendant of `AppShell` should use `useProfile()` instead of querying `auth.getUser()`/`profiles`/`tenants` itself — every `*Client.tsx` component, Header, Sidebar, and DashboardStats/Charts already do this. `profile.tenant` holds the full tenants row (all columns) so consumers needing business info (name/phone/address/logo/license_number/tax_id for invoices etc.) read it off `profile.tenant` instead of querying tenants again.
- **AppShell/Provider tree gotcha:** `useProfile()` only works in components rendered as JSX *children* of `<AppShell>`. A component that wraps itself in `<AppShell>...</AppShell>` in its own return statement (rather than being passed as AppShell's children from its page.tsx) CANNOT call `useProfile()` at its own top level — the Provider is a descendant of that component, not an ancestor, so context resolves to the default (null/loading forever). Fix: split into an outer wrapper (does the `<AppShell>` wrapping, no hooks needed) and an inner component (calls `useProfile()`, rendered as AppShell's child). This bit `ExpensesClient.tsx` and `app/dashboard/page.tsx` — both fixed by extracting an inner content component.

## Decision Log

<!-- Significant technical decisions with rationale. Why X was chosen over Y. -->
- **[2026-07-05] Financial module overhaul (expenses/debts/billing):** Implemented approved plan `lively-imagining-tarjan.md`.
  - `expenses`: added `recurring_expense_id` (back-ref to `recurring_expenses` template) and `amortize_months` (nullable smallint) for manually-entered annual expenses that should be reported as prorated across N months while still showing in full on their real payment date in the normal ledger view. `ExpensesClient.tsx` summary tab has an "actual vs. prorated" toggle (`sumMode`) driving this; the main monthly tab got a category-grouped accordion (`groupedView`/`openCats`) as an alternative to the flat list.
  - Supplier checks/debts: rejected FIFO (see User Preferences). Added `supplier_debt_payments` junction table recording exactly which check/payment settled which debt, populated by `lib/debts/reconcileSupplierPayment.ts` from an **explicit** allocation list chosen in the UI (checkbox list of open debt-months per supplier, in `ScheduledPaymentsModal.tsx` and `DebtsClient.recordPayment`). This replaced the old fragile heuristic at `DebtsClient.tsx` (`due_date` month === debt month) which never actually linked checks to debts correctly. `scheduled_payments` gained `check_number` and `series_id` (for the new check-series batch-creation feature — supports equal split or "round amount + remainder on first/last check"). `DebtsClient.tsx` gained a `calendar` tab: all unpaid `scheduled_payments` across every supplier, grouped by due-month, for cash-flow forecasting.
  - Billing: found the item/entry VAT toggle (`VatToggle`, before/after) was **cosmetic only** — never affected the saved `amount`. Fixed at save time (`saveItem`/`saveEntry` in `BillingClient.tsx`): if mode is `'before'` (pre-VAT figure entered), `amount = withVat(raw)`; only applies to `type==='fixed'` (rent/arnona), never to `type==='meter'` (electricity has no VAT in this sublet arrangement, confirmed by user). Added `billing_entries.period_start/period_end` (informational, for irregular ~2-month electric meter readings) and `billing_items/entries.fixed_addon` (electric company's fixed infrastructure fee on top of kWh usage — stays on the payer's side, never charged to the sub-tenant).
- **[2026-07-06] Supplier debt detail moved to a new dedicated page** (same plan file, second round): the whole detailed supplier-debt UI that had just been built into `DebtsClient.tsx`'s suppliers tab (month-grouped invoice cards, the checks calendar, add/edit) was **relocated** into a brand-new page `app/supplier-tracking/page.tsx` + `components/supplier-tracking/SupplierTrackingClient.tsx`, reachable via a new sidebar item (`Sidebar.tsx` `NAV_ITEMS`/`ICONS`/`SECTIONS`, href `/supplier-tracking`, reuses the existing `module: 'suppliers'` permission). **`DebtsClient.tsx`'s suppliers tab is now summary-only** (supplier name + phone + final balance, a link out — no add/edit/detail, no calendar tab; `Tab` type is back to `'customers'|'suppliers'|'summary'`). If asked to touch supplier debt detail/checks/invoices UI, the code lives in `SupplierTrackingClient.tsx` now, not `DebtsClient.tsx`.
  - Also added `direction: 'charge'|'credit'` to `supplier_debts` (migration `064_supplier_debt_direction.sql`) to support supplier credit notes (returned goods/discounts) that reduce the balance, alongside regular charges. `bal()` (defined independently in `DebtsClient.tsx`, `SuppliersClient.tsx`, and `SupplierTrackingClient.tsx` — no shared util) became direction-aware: `direction==='credit' ? -amount : max(0, amount-paid)`, which automatically fixes every existing sum built on top of it. Credit rows are inserted with `paid:0, is_closed:true` immediately (never "paid" by a check).
  - New entries in `SupplierTrackingClient.tsx` are **one independent `supplier_debts` row per invoice/credit line** (own date, own direction) via bulk-insert — no more bundling multiple invoices under one shared date in the `invoices` jsonb array. Editing an existing (possibly legacy-bundled) row still does a single-row update, preserving old data untouched. The by-month view renders a real `<table>` (מספר חשבונית/תאריך/סוג/סכום) with a totals row (חיוב/זיכוי/נטו/שולם/יתרה) instead of the old `InvoiceCell` chip cards.
